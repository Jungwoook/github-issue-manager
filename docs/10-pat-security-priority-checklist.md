# PAT 보안 포인트 우선순위 정리

## 1. 목적

이 문서는 현재 프로젝트의 PAT(Personal Access Token) 처리 구조를 기준으로,
프론트엔드와 백엔드에서 어떤 보안 포인트를 어떤 순서로 수정해야 하는지 정리한다.

정리 기준은 아래와 같다.

1. 토큰 탈취 가능성을 얼마나 빠르게 낮출 수 있는가
2. 구현 난이도 대비 효과가 큰가
3. 현재 구조를 크게 깨지 않고 바로 적용 가능한가

---

## 2. 현재 구조 요약

현재 PAT 흐름은 아래와 같다.

1. 프론트엔드에서 사용자가 PAT를 입력한다.
2. 프론트엔드가 `POST /api/github/token` 으로 `accessToken` 문자열을 전송한다.
3. 백엔드가 GitHub `/user` API로 토큰을 검증한다.
4. 검증 성공 시 PAT를 암호화해 `github_accounts.access_token_encrypted` 에 저장한다.
5. 이후 저장소/이슈/댓글 API 호출 시 복호화해서 GitHub API 호출에 사용한다.

이 구조에서 핵심 리스크는 다음 두 가지다.

- 전송 전후의 평문 PAT 노출
- 저장된 PAT 복호화 및 재사용 과정의 방어 수준 부족

---

## 3. 수정 우선순위

## 3.1 1순위: 즉시 적용해야 하는 항목

이 항목들은 구현 난이도가 낮고, 바로 위험을 줄일 수 있다.

### 1. 프론트 입력 필드를 `password` 타입으로 변경

- 현재 `GitHubTokenForm` 의 PAT 입력창은 일반 텍스트 입력처럼 동작한다.
- 화면 공유, 주변 시선, 브라우저 자동 캡처, 디버깅 과정에서 토큰이 그대로 노출될 수 있다.

수정 포인트:

- `frontend/src/widgets/github-token/GitHubTokenForm.tsx`
- `type="text"` 를 `type="password"` 로 변경
- 필요하면 "보기/숨기기" 토글은 별도 버튼으로 제공

이유:

- 네트워크 보안과 별개로, 가장 쉽게 줄일 수 있는 노출 경로다.

### 2. 운영 환경에서 HTTPS 강제

- 프론트가 문자열로 PAT를 보내는 것 자체보다 더 중요한 것은 전송 구간 보호다.
- HTTP 환경에서는 PAT가 네트워크 구간에서 탈취될 수 있다.

수정 포인트:

- 운영 프론트 URL과 백엔드 API URL을 모두 HTTPS로만 제공
- 리버스 프록시 또는 LB 레벨에서 HTTP -> HTTPS 리다이렉트 적용
- 문서와 배포 체크리스트에 HTTPS 필수 명시

이유:

- 현재 구조에서는 TLS가 사실상 필수 보안 조건이다.

### 3. 로그에 PAT가 남지 않도록 점검

- 요청 바디, 예외 로그, 디버그 로그, 프록시 로그에 PAT가 남으면 암호화 저장 여부와 무관하게 사고가 된다.

수정 포인트:

- 백엔드 request logging 필터 또는 access log 설정 점검
- 프론트에서 에러 객체 전체를 콘솔 출력하는 코드 금지
- API 게이트웨이, 프록시, APM, 에러 수집 도구에서 request body 수집 여부 점검

이유:

- 실제 운영 사고는 저장소보다 로그에서 더 자주 발생한다.

### 4. 프론트 메모리 보관 시간을 최소화

- 현재 PAT는 프론트 state에 저장됐다가 등록 성공 시 초기화된다.
- 실패 시에는 값이 계속 남아 있을 수 있다.

수정 포인트:

- 등록 성공 시 즉시 초기화 유지
- 등록 실패 시에도 재시도 UX를 해치지 않는 선에서 명시적으로 초기화할지 검토
- 브라우저 영구 저장소(`localStorage`, `sessionStorage`)에는 절대 저장하지 않기

이유:

- 클라이언트 메모리 노출 시간을 줄이는 것은 기본 방어다.

## 3.2 2순위: 백엔드에서 빠르게 강화해야 하는 항목

이 항목들은 실제 토큰 저장과 복호화 사용 구조를 강화한다.

### 5. PAT 암호화 방식을 `AES-GCM` 기반으로 개선

- 현재 구현은 `Cipher.getInstance("AES")` 를 사용한다.
- 이 방식은 provider 기본 설정에 의존하며, 일반적으로 안전한 최신 구성이라고 보기 어렵다.
- 무결성 보호까지 포함하는 AEAD 방식이 아니다.

수정 포인트:

- `backend/src/main/java/com/jw/github_issue_manager/service/PatCryptoService.java`
- `AES/GCM/NoPadding` 사용
- 암호화 시 랜덤 IV 생성
- `IV + ciphertext + tag` 저장 구조로 변경

이유:

- 지금 구조의 가장 직접적인 백엔드 보안 개선 포인트다.

### 6. 키 파생 방식을 단순 SHA-256 절단 방식에서 개선

- 현재는 `GITHUB_PAT_ENCRYPTION_KEY` 문자열을 SHA-256으로 해싱한 뒤 앞 16바이트를 AES 키로 사용한다.
- 동작은 단순하지만 키 강화 관점에서는 충분히 좋은 방식이 아니다.

수정 포인트:

- KDF 사용 검토: PBKDF2, scrypt, Argon2 계열
- 최소한 키 길이와 형식을 명확히 정의
- 운영 시 랜덤하고 긴 키를 사용하도록 배포 규칙 문서화

이유:

- 키 생성이 약하면 저장 암호화 수준도 함께 낮아진다.

### 7. PAT 복호화 실패와 GitHub 인증 실패를 구분해서 처리

- 현재는 복호화 실패, 잘못된 키, 만료/권한 부족 토큰이 모두 비슷하게 보일 수 있다.
- 운영 시 원인 파악이 어려워지고, 사용자 경험도 나빠진다.

수정 포인트:

- 복호화 실패 예외와 GitHub API 인증 실패를 구분
- 사용자 메시지는 일반화하되, 내부 로그는 원인 카테고리로 분리
- 필요하면 토큰 재등록 유도

이유:

- 보안 사고 대응과 운영 디버깅 품질이 올라간다.

### 8. PAT 사용 시점의 검증 실패 대응 강화

- 현재 `tokenVerifiedAt` 은 등록 시점에만 갱신된다.
- 이후 토큰 만료, 권한 변경, 폐기 여부는 GitHub API 실패를 통해서만 드러난다.

수정 포인트:

- GitHub 401/403 응답 시 토큰 상태 재확인 로직 검토
- 연속 실패 시 "재연결 필요" 상태 제공
- `lastAuthenticatedAt`, `tokenVerifiedAt` 활용 전략 재정의

이유:

- 저장된 PAT가 계속 유효하다고 가정하는 것은 운영상 취약하다.

## 3.3 3순위: 프론트/백엔드 경계 보강

이 항목들은 직접적인 토큰 저장 문제보다, 토큰 탈취 경로를 줄이는 데 중요하다.

### 9. XSS 방어 점검

- 프론트에서 PAT를 입력받는 구조에서는 XSS가 사실상 토큰 탈취 경로가 된다.
- 브라우저에서 실행되는 악성 스크립트는 입력값을 읽을 수 있다.

수정 포인트:

- 사용자 입력 렌더링 지점 점검
- `dangerouslySetInnerHTML` 사용 여부 점검
- CSP(Content Security Policy) 적용 검토
- 서드파티 스크립트 최소화

이유:

- 프론트에서 토큰을 받는 구조라면 XSS 방어 수준이 곧 토큰 보호 수준이다.

### 10. CORS와 세션 쿠키 정책 재점검

- 현재 프론트는 `credentials: 'include'` 를 사용한다.
- 이 구조에서는 허용 origin, 쿠키 속성, 세션 정책이 정확해야 한다.

수정 포인트:

- 허용 origin을 운영 도메인으로 제한
- `SameSite`, `Secure`, `HttpOnly` 쿠키 정책 확인
- 개발/운영 환경별 CORS 설정 분리

이유:

- 세션 탈취나 잘못된 cross-origin 허용은 간접적으로 PAT 사용 권한 오남용으로 이어질 수 있다.

### 11. CSRF 관점 점검

- 세션 기반 인증을 사용하면 CSRF 방어 여부를 확인해야 한다.
- 특히 토큰 연결/해제, 이슈 생성/수정/삭제 같은 상태 변경 요청은 영향이 크다.

수정 포인트:

- Spring Security 사용 여부와 CSRF 설정 확인
- SameSite 쿠키 정책과 함께 실제 방어가 되는지 검토
- 필요하면 CSRF 토큰 또는 stricter origin 검증 적용

이유:

- 세션 인증 구조에서는 빠뜨리기 쉬운 기본 점검 항목이다.

## 3.4 4순위: 구조 개선 항목

이 항목들은 바로 막아야 하는 문제라기보다, 장기적으로 PAT 직접 처리 구조를 줄이는 방향이다.

### 12. 토큰 회전과 운영 키 변경 전략 수립

- 암호화 키가 바뀌면 기존 PAT 복호화가 불가능해질 수 있다.
- 현재는 키 회전 전략이 없다.

수정 포인트:

- 키 버전 관리 도입 검토
- 기존 데이터 재암호화 절차 정의
- 운영 키 변경 시 영향 범위 문서화

이유:

- 운영 단계에서 실제로 문제를 일으키는 항목이다.

### 13. 연결 해제 시 캐시 정리 정책 명확화

- 현재 연결 해제는 PAT 제거와 세션 종료 중심이다.
- 과거 캐시 데이터는 남아 있을 수 있다.

수정 포인트:

- `repository_caches`, `issue_caches`, `comment_caches` 유지 정책 정의
- 계정 연결 해제 시 데이터 잔존 여부를 문서화
- 필요하면 "연결 해제 + 캐시 정리" 옵션 제공

이유:

- 직접적인 토큰 보호보다 개인정보 및 데이터 관리 정책 측면에서 중요하다.

### 14. 장기적으로 OAuth 또는 GitHub App 방식 전환 검토

- 사용자가 PAT를 직접 입력하는 구조는 본질적으로 프론트에서 민감정보를 받는다.
- 이 구조는 구현은 빠르지만, 장기 운영 보안 모델로는 불리하다.

수정 포인트:

- OAuth App 또는 GitHub App 기반 아키텍처 검토
- 토큰 범위, 수명, 회수 정책을 플랫폼 모델에 맞게 재설계

이유:

- 가장 근본적인 구조 개선 방향이다.

## 4. 권장 실행 순서

실제 작업 순서는 아래처럼 가져가는 것이 효율적이다.

### Step 1. 프론트 즉시 수정

- PAT 입력창 `password` 전환
- 콘솔/에러 출력 점검
- 브라우저 저장소 미사용 원칙 재확인

### Step 2. 운영 설정 보강

- HTTPS 강제
- CORS origin 제한
- 세션 쿠키 `Secure`, `HttpOnly`, `SameSite` 점검

### Step 3. 백엔드 저장 보안 강화

- `PatCryptoService` 를 `AES-GCM` 기반으로 교체
- 키 관리 방식 강화
- 복호화 실패 / 인증 실패 처리 구분

### Step 4. 요청 경계 보강

- XSS 점검
- CSRF 점검
- 로그/관측 도구의 request body 수집 차단

### Step 5. 구조 개선

- 키 회전 정책
- 연결 해제 시 데이터 처리 정책
- OAuth / GitHub App 전환 검토

---

## 5. 우선순위 요약

가장 먼저 해야 할 항목만 짧게 요약하면 아래와 같다.

1. PAT 입력창을 `password` 로 바꾼다.
2. 운영 환경에서 HTTPS를 강제한다.
3. 로그와 관측 도구에 PAT가 남지 않게 막는다.
4. `PatCryptoService` 를 `AES-GCM` 기반으로 교체한다.
5. XSS, CORS, 세션 쿠키 정책을 함께 점검한다.

이 다섯 가지를 먼저 정리하면, 현재 PAT 기반 구조에서 가장 큰 보안 구멍을 우선적으로 줄일 수 있다.
