# 260421 Task

## 1. 플랫폼 모듈화 이후 구조 문서 작성
요약: registry / capability / metadata 기반 목표 구조 정리

- 문제: 현재 구조는 어댑터 분리 단계이며 새 플랫폼 추가 시 공통 코드 수정 발생
- 지시사항: 현재 문서를 기반으로 모듈화 이후 파일 구조와 동작 방식을 그래프 포함 문서화
- 작업: 모듈 경계, 목표 파일 구조, 핵심 계약, 백엔드/프론트 동작 그래프 작성
- 결과: `docs/13-platform-modularization-design.md` 추가, 전환 순서와 회귀 방지 기준 확보

## 2. 모듈러 모놀리스 관점 보정
요약: 단순 패키지 분리에서 업무 모듈 경계 중심으로 재작성

- 문제: 기존 초안은 플랫폼 어댑터 등록 구조 중심이라 모듈러 모놀리스로 보기 어려움
- 지시사항: 모듈 간 의존성을 줄이는 모듈러 모놀리스 방식으로 재정의
- 작업: app / platform / connection / repository / issue / comment / shared 모듈 경계와 금지 의존 정리
- 결과: 공개 API 패키지, 내부 구현 은닉, 의존 규칙 테스트 기준 반영

## 3. Gradle 멀티 모듈 전환 순서 명시
요약: 패키지 기반 안정화 후 Gradle 물리 모듈로 승격

- 문제: 최종 멀티 모듈 목표와 패키지 기반 중간 단계의 관계가 불명확
- 지시사항: 최종은 Gradle 기반 멀티 모듈, 전환은 패키지 기반 모듈 단계를 거치도록 명시
- 작업: 1차 패키지 구조, 최종 Gradle 구조, 모듈 의존 방향, 승격 단계 추가
- 결과: 점진 전환 순서와 컴파일 기반 의존 차단 목표 정리

## 4. 전환 순서 사유 보강
요약: 단계별 작업 이유 추가

- 문제: 전환 순서에 작업 목적은 있으나 순서의 이유가 부족
- 지시사항: 각 단계별로 왜 해당 순서로 진행하는지 설명 추가
- 작업: 공개 API, connection, platform, 업무 모듈, 의존 테스트, Gradle 승격 단계별 이유 작성
- 결과: 논리 경계 안정화 후 물리 모듈 승격이라는 전환 근거 명확화

## 5. 전환 단계별 설명 보강
요약: 단계별 산출 상태와 다음 단계 연결 설명 추가

- 문제: 전환 순서가 작업 항목 중심이라 실제 변화 상태를 이해하기 어려움
- 지시사항: 각 단계마다 어떤 상태를 만들고 왜 다음 단계로 이어지는지 설명 추가
- 작업: 공개 API, 연결, 플랫폼, 업무 캐시, 의존 테스트, Gradle 골격, 의존성, Spring/JPA scan, 컴파일 경계 설명 보강
- 결과: 전환 순서의 의도와 단계별 완료 상태 이해 가능
