# 260417 Task

## 1. GitHub/GitLab 비교 문서 작성
요약: GitLab 추가 전 플랫폼 적용 차이 비교 문서화
- 문제: GitLab 추가 전 인증, 식별자, 이슈/댓글 API 차이를 먼저 정리할 필요
- 지시사항: 공식 문서 기준으로 GitHub와 GitLab 적용 차이 비교
- 작업: 토큰 인증, 사용자 확인, 저장소/프로젝트, 이슈, 댓글, 운영 차이 비교 문서 작성
- 결과: GitLab 추가 전 검토 기준 정보와 구현 선행 판단 근거 정리

## 2. GitLab 추가 선행 이슈 정리
요약: 현재 구조에서 GitLab 추가 시 먼저 보완할 부분 정리
- 문제: 기능 유사성 외에 project path, issue iid, base URL 차이가 구현 리스크로 존재
- 지시사항: 현재 공통 구조에 미치는 영향과 보완 지점 분리 정리
- 작업: 공통 사용 가능 요소, 선행 보완 요소, 권장 구현 순서 정리
- 결과: GitLab 추가 착수 전 설계 체크리스트 정보

## 3. GitLab 연동 설계 계획 작성
요약: 비교 문서 기반으로 GitLab 추가 설계 계획 문서 작성
- 문제: GitLab 추가 전 구현 범위, 책임 분리, 식별자 규칙을 먼저 고정할 필요
- 지시사항: GitHub/GitLab 비교 문서를 기준으로 구현 순서와 설계 원칙 정리
- 작업: 목표 범위, 백엔드 구조, API 계약, 프론트 반영, 테스트 계획, 단계별 순서 문서화
- 결과: GitLab 1차 추가를 위한 설계 계획과 구현 체크포인트 정리

## 4. GitLab 1차 범위 제한
요약: 범위 확장 방지를 위한 1차 구현 경계 정의
- 문제: self-managed, 고급 토큰, GitLab 전용 기능을 함께 넣으면 초기 구조 검증이 어려움
- 지시사항: 최소 기능 범위와 후속 과제를 분리해 설계 계획에 반영
- 작업: GitLab.com 기준, 프로젝트/이슈/댓글 중심, 공통 UI 유지 원칙 정리
- 결과: 1차 목표와 후속 확장 범위가 분리된 설계 기준 확보

## 5. GitLab 백엔드 1차 구현
요약: 공통 게이트웨이 구조에 GitLab 어댑터와 인증 메타데이터 추가
- 문제: 두 번째 플랫폼을 실제로 붙여 공통 포트 구조와 백엔드 계약이 유지되는지 검증 필요
- 지시사항: GitLab.com 기준 사용자 검증, 프로젝트/이슈/댓글 API를 공통 게이트웨이에 연결
- 작업: `PlatformType` 확장, `GitLabApiClient` / `GitLabPlatformGateway` 구현, 플랫폼별 token scope 처리, GitLab 매핑 테스트 추가
- 결과: 프론트 변경 없이도 GitLab 토큰 등록과 프로젝트/이슈/댓글 백엔드 연동 기반 확보

## 6. GitLab 2차 범위 정의
요약: GitLab 실제 사용 흐름 완성을 위한 2차 구현 범위 정리
- 문제: 1차는 백엔드 최소 연동까지라서 프론트 연결, base URL, 프로젝트 식별 규칙, 접근 제어 보강이 남아 있음
- 지시사항: 1차 완료 이후 필요한 계약 확장과 구현 범위를 GitLab 설계 문서에 분리 정리
- 작업: `baseUrl`, `path_with_namespace`, `iid`, 프론트 연결, 예외 메시지, 테스트 보강을 2차 범위로 명시
- 결과: GitLab 후속 구현의 목표, 제외 범위, 완료 기준이 분리된 2차 작업 기준 확보

## 7. GitLab 2차 구현
요약: GitLab 연결 계약과 프론트 실제 사용 흐름 보강
- 문제: 1차 백엔드 연동만으로는 GitLab 선택, base URL 설정, 실제 연결 화면 흐름이 완결되지 않음
- 지시사항: 연결별 `baseUrl` 계약을 추가하고 프론트에서 GitHub/GitLab 전환과 GitLab 입력 흐름을 반영
- 작업: `PlatformConnection`에 `baseUrl` 추가, 게이트웨이 per-connection base URL 사용, 플랫폼 탭 UI와 GitLab 연결 폼 확장
- 결과: GitLab.com 기본 연결과 self-managed 확장 포인트를 포함한 2차 사용자 흐름 기반 확보
