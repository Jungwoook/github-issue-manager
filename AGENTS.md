# AGENTS.md

## 목적

이 문서는 이 저장소에서 Codex를 포함한 코딩 에이전트가 작업할 때 따라야 할 기본 작업 원칙을 정리한다.

- 변경은 가능한 한 작고 검토 가능하게 유지한다.
- 사용자가 명시적으로 요청하지 않은 동작 변경은 피한다.
- 기존 GitHub 기능은 플랫폼 공통화 작업 중에도 안정적으로 유지한다.

## 기본 작업 원칙

- 구조 변경 전에 기존 `docs/` 문서를 먼저 확인한다.
- 작업 중 발견한 규칙이나 결정사항은 이후 작업과 충돌하지 않도록 문서에 반영한다.
- 큰 리팩토링은 한 번에 끝내려 하지 말고, 동작을 유지하는 중간 단계를 만든다.
- 네이밍 변경보다 의존성 분리와 인터페이스 추상화를 우선한다.
- 사용자가 별도로 요청하지 않으면 불필요한 범위 확장은 하지 않는다.

## 문서 작업 규칙

- 의미 있는 설계 변경, 구조 변경, 작업 흐름 변경은 `docs/`에 반영한다.
- 작업 기록은 `docs/task/YYYYMMDD.md` 형식으로 남긴다.
- 작업 기록은 작업을 진행하는 오늘 날짜 기준 파일에 작성하고, 날짜가 바뀌면 새 파일을 생성한다.
- 작업 기록 문서는 `제목 + 요약 + 문제 + 지시사항 + 작업 + 결과` 형식으로 작성한다.
- 각 항목은 서술형보다 개조식 중심으로, 핵심어만 남기듯 짧게 작성한다.
- 한 줄에 문장을 길게 풀지 말고 `대상 / 조치 / 결과` 수준으로 압축한다.
- 예시
  `## 1. 플랫폼 공통화 1차 리팩토링`
  `요약: 서비스 → 공통 게이트웨이 구조로 전환`
  `- 문제: GitHub API 타입 직접 의존`
  `- 지시사항: 공통 포트 기준 1차 리팩토링`
  `- 작업: Port / Remote DTO / Gateway 도입`
  `- 결과: 기존 기능 유지 + 공통화 기반 확보`
- 파일명, 세부 경로, 장황한 구현 설명보다 작업 의도와 결과 중심으로 정리한다.

## 플랫폼 공통화 작업 원칙

- 플랫폼 공통화는 `공통 코어 + GitHub 어댑터` 구조를 목표로 한다.
- 1차 작업은 백엔드 공통 인터페이스 분리부터 진행한다.
- 서비스 레이어는 GitHub 구현체가 아니라 공통 포트에 의존하도록 바꾼다.
- GitHub 전용 식별자는 장기적으로 `platform + externalId` 구조로 일반화한다.
- 현재 동작 중인 GitHub 흐름을 깨지 않는 방향으로 단계적으로 전환한다.

## 백엔드 작업 원칙

- API와 도메인 구조가 함께 바뀌는 작업은 백엔드부터 정리한다.
- 플랫폼 확장 리팩토링에서는 인터페이스, DTO, 게이트웨이 분리를 먼저 진행한다.
- 기존 구현을 바로 제거하지 말고, 공통 인터페이스 뒤로 이동시키는 방식을 우선한다.
- 백엔드 동작 변경이 있으면 가능한 범위에서 테스트로 검증한다.

## 프론트 작업 원칙

- 플랫폼 공통화 작업에서 프론트는 백엔드 공통 모듈 분리가 완료된 이후 한 번에 수정한다.
- 중간 단계에서 프론트 API 모듈, 라우트, 쿼리 키를 반복적으로 변경하지 않는다.
- 프론트 수정은 백엔드 계약이 안정화된 뒤 일괄 정리하는 것을 기본 원칙으로 한다.
- 프론트는 기능 동작을 우선 확인하고, 사용자가 별도로 요청하지 않으면 구조나 코드 품질 리뷰 범위에 포함하지 않는다.

## 리뷰 작업 원칙

- 리뷰는 백엔드 구조와 설계를 중심으로 진행한다.
- 프론트는 기능 동작 여부만 확인하고, 구조, 코드 스타일, 컴포넌트 분리 같은 항목은 리뷰 대상에서 제외한다.
- 백엔드 모듈 경계, 인터페이스 의존성, 플랫폼 공통화 설계, 기존 GitHub 흐름 영향 여부를 우선 확인한다.

## 커밋 원칙

- 커밋은 의미 단위로 나눈다.
- 문서 변경은 가능하면 문서끼리 묶어서 커밋한다.
- 구조 리팩토링 커밋과 동작 변경 커밋은 가능하면 분리한다.
- 커밋 메시지는 작업 의도가 드러나도록 간결하게 작성한다.
- 사용자가 커밋을 지시하면 커밋에서 멈추지 않고, 푸시와 PR 생성까지 이어서 진행한다.
- PR 제목과 본문은 한글로 작성한다.
- 권한 요청이 필요한 git 작업은 가능한 한 고정된 명령 패턴으로 수행해 승인 재사용이 가능하도록 한다.
- `git add`, `git commit`, `git switch`, `git push` 같은 Git 쓰기 명령은 가능한 한 처음부터 권한 상승 모드로 실행한다.
- Git 쓰기 명령은 일반 권한으로 먼저 시도한 뒤 재시도하지 말고, 승인된 패턴이 있으면 우선 사용한다.

## 우선 확인 사항

작업을 시작할 때 아래 항목을 먼저 확인한다.

- 이미 관련 설계 문서가 있는지
- 같은 주제의 작업 기록이 `docs/task/`에 있는지
- 이번 작업이 백엔드 우선인지, 프론트까지 포함하는지
- 기존 GitHub 흐름에 영향이 있는지

## 권장 작업 순서

1. 기존 문서와 관련 코드를 확인한다.
2. 변경 범위를 작게 나눌 수 있는지 판단한다.
3. 필요하면 설계 문서를 먼저 갱신한다.
4. 백엔드 공통화 작업을 우선 적용한다.
5. 검증 가능한 변경은 테스트나 빌드로 확인한다.
6. 작업 기록 문서를 갱신한다.

## 피해야 할 작업 방식

- 프론트와 백엔드를 동시에 크게 뒤집는 변경
- GitHub 전용 구조를 충분한 중간 단계 없이 한 번에 제거하는 변경
- 문서 없이 진행되는 구조 리팩토링
- 지시사항과 결과가 추적되지 않는 작업 기록
